# Database

SQLite file at `data.db` in the project root. WAL mode is enabled so the server can read while writing, and external tools (DB Browser, `sqlite3` CLI) can open the file while the server is running — they will see `data.db` + `data.db-wal`.

All DB access goes through `db.js`. Never `import Database` outside that file.

---

## Schema

### `users`

```sql
CREATE TABLE users (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  username      TEXT    UNIQUE NOT NULL,
  password_hash TEXT    NOT NULL,
  role          TEXT    NOT NULL DEFAULT 'user'
                        CHECK(role IN ('admin', 'user'))
);
```

Passwords are hashed with `bcrypt` (cost factor 10). Plaintext passwords are never stored.

---

### `user_instances`

```sql
CREATE TABLE user_instances (
  user_id     INTEGER NOT NULL,
  instance_id TEXT    NOT NULL,
  PRIMARY KEY (user_id, instance_id),
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

CREATE INDEX idx_user_instances_user     ON user_instances(user_id);
CREATE INDEX idx_user_instances_instance ON user_instances(instance_id);
```

Junction table: which users have access to which instances. Admins implicitly access all instances without rows here.

---

### `instance_api_keys`

```sql
CREATE TABLE instance_api_keys (
  instance_id TEXT PRIMARY KEY,
  api_key     TEXT NOT NULL UNIQUE
);
```

64-character hex strings generated by `crypto.randomBytes(32).toString('hex')`. One key per instance; regenerating replaces the row.

---

### `instance_messages`

```sql
CREATE TABLE instance_messages (
  id                INTEGER PRIMARY KEY AUTOINCREMENT,
  instance_id       TEXT    NOT NULL,
  message_id        TEXT    NOT NULL,
  from_jid          TEXT    NOT NULL,
  sender_display    TEXT    NOT NULL,
  sender_number     TEXT,
  group_name        TEXT,
  body              TEXT    NOT NULL,
  source            TEXT    NOT NULL DEFAULT 'native',
  message_kind      TEXT    NOT NULL DEFAULT 'individual',
  message_timestamp INTEGER,
  created_at        INTEGER DEFAULT (strftime('%s','now')),
  UNIQUE(instance_id, message_id)
);

CREATE INDEX idx_instance_messages_instance ON instance_messages(instance_id);
CREATE INDEX idx_instance_messages_created  ON instance_messages(instance_id, created_at DESC);
```

**Column notes:**

| Column | Notes |
|---|---|
| `message_id` | WhatsApp message ID string. `UNIQUE(instance_id, message_id)` prevents duplicate inserts. |
| `from_jid` | Full JID — `number@c.us` for users, `groupId@g.us` for groups. |
| `sender_display` | Best human-readable label: push name, contact name, or number. Falls back to `from_jid`. |
| `sender_number` | Digits only (no `@` suffix). `NULL` for group JIDs that can't be resolved. |
| `group_name` | Group chat name; `NULL` for individual messages. |
| `source` | `'native'` (received by local WA client) or `'webhook'` (via POST webhook). |
| `message_kind` | `'individual'`, `'group'`, or `'status'`. Derived from `from_jid` and `isStatus`. |
| `message_timestamp` | Unix seconds from WhatsApp. Webhook messages use server time. |
| `created_at` | Unix seconds when the row was inserted. |

---

### `instance_linked_accounts`

```sql
CREATE TABLE instance_linked_accounts (
  instance_id  TEXT PRIMARY KEY,
  account_jid  TEXT,
  account_number TEXT,
  account_name TEXT,
  updated_at   INTEGER DEFAULT (strftime('%s','now'))
);
```

Stores the WhatsApp number linked to each instance. Updated on each `ready` event via `syncInstanceLinkedAccount()`. Used to detect when an instance is re-authenticated with a different number (`linkedAccountChanged`).

## Indexes

| Index | Table | Purpose |
|---|---|---|
| `idx_user_instances_user` | `user_instances` | Lookup all instances for a user |
| `idx_user_instances_instance` | `user_instances` | Lookup all users for an instance |
| `idx_instance_messages_instance` | `instance_messages` | Filter by instance (all queries) |
| `idx_instance_messages_created` | `instance_messages` | `ORDER BY created_at DESC` query (paginated log) |

## Migration Pattern

`initDb()` in `db.js` runs on every startup:

1. `CREATE TABLE IF NOT EXISTS` — idempotent table creation.
2. `ensureColumn(db, table, column, definition)` — adds a column if it doesn't exist (SQLite's `ALTER TABLE ADD COLUMN`). No-ops if already present.
3. Backfill functions — one-time data migrations that run in-process:
   - `backfillMessageKinds` — sets `message_kind` for rows that have `NULL` (pre-migration rows).
   - `backfillSenderNumbers` — extracts `sender_number` from `from_jid` where missing.
   - `backfillGroupNames` — (reserved; currently a no-op placeholder).
4. `seedAdminIfNeeded` — creates the default `admin` user if the `users` table is empty.

**Adding a new column:** call `ensureColumn()` in `initDb()` and add a backfill function if existing rows need a default computed value. Do not add `NOT NULL` columns without a `DEFAULT` — SQLite rejects that on `ALTER TABLE ADD COLUMN`.

## Key DB Functions (`db.js`)

```
initDb()                         Boot-time setup: schema + migrations + seed
getUserByUsername(username)      Login lookup (includes password_hash)
getUserById(id)                  Session rehydration (excludes password_hash)
verifyLogin(username, password)  bcrypt compare; returns user or null
getAllUsers()                     Admin list
createUser(username, pw, role)   bcrypt hash + INSERT
userCanAccessInstance(id, role, instanceId)  true if admin or has user_instances row
getInstanceIdsForUser(id, role)  All instances accessible to a user
assignInstanceToUser(uid, iid)   INSERT OR IGNORE into user_instances
removeInstanceFromUser(uid, iid) DELETE from user_instances
updateUserPassword(uid, pw)      bcrypt hash + UPDATE
deleteUser(uid)                  DELETE (cascades user_instances)
getAdminCount()                  Guard against deleting the last admin
setInstanceApiKey(iid, key)      INSERT OR REPLACE
getInstanceApiKey(iid)           Returns { api_key } or undefined
getInstanceIdByApiKey(key)       Reverse lookup for auth middleware
deleteInstanceApiKey(iid)        Called on instance delete
getInstanceLinkedAccount(iid)    Returns stored account or null
setInstanceLinkedAccount(iid, …) UPSERT linked account
deleteInstanceLinkedAccount(iid) Called on instance delete
insertMessage(…)                 INSERT OR IGNORE (dedup on instance_id+message_id)
getMessagesForInstance(iid, n)   SELECT ordered by created_at DESC LIMIT n
deleteMessagesForInstance(iid)   DELETE all messages; returns affected row count
```

## Inspecting the Database

```bash
# Stop the server first (or use a WAL-aware tool)
sqlite3 data.db ".tables"
sqlite3 data.db "SELECT id, username, role FROM users;"
sqlite3 data.db "SELECT instance_id, COUNT(*) FROM instance_messages GROUP BY instance_id;"
```

GUI tools (DB Browser for SQLite, TablePlus) can open `data.db` while the server runs — they read from the WAL automatically.

WAL checkpoint runs on startup (`PRAGMA wal_checkpoint(TRUNCATE)`) to keep the main file current.
